# 数据、隐私与公开边界

## 1. 数据分层

| 层级 | 示例 | 允许公开 |
|---|---|---|
| 原始临床数据 | Excel、姓名、登记号、签名、原始路径 | 否 |
| 内部结构化数据 | `patient_records/*.json`、内部 bundle | 否 |
| 研究冻结数据 | analysis-ready 快照、字段字典 | 默认否，按伦理和数据治理决定 |
| 匿名化演示数据 | `dashboard_ui/demo_data/data_bundle.js` | 仅经审计构建后 |
| 发布产物 | `data/index.json`、`data/records/Pxxxx.json` | 审计通过后 |

## 2. 当前脱敏策略

发布构建器执行以下处理：

- 患者改为 `P0001` 形式的稳定别名。
- 姓名改为“患者 Pxxxx”。
- 年龄转换为十岁年龄段。
- 日期按患者稳定偏移，保留纵向间隔而隐藏真实日期。
- 删除原始路径、文件名、来源元数据和签名字段。
- 扫描并替换已知姓名、登记号、电话、证件号、邮箱和本地路径。
- `备注`、`说明`、`建议`、`病史`等叙述字段继续不公开。
- 本地隐私策略 `2.1.0` 保留结论字段，但仍执行直接标识替换和全包泄露扫描。

## 3. 结论字段的当前决策

2026-07-10，用户明确希望公开经直接标识脱敏后的真实程控结论，理由是 `AS-VP`、`VS`、`起搏器功能未见异常`等属于标准医学/设备描述。

当前状态：

- 本地代码已按该要求保留结论。
- 本地审计通过；943 条结论均不再是“不发布”占位文本。
- 生产上传因外部安全策略阻止敏感病例级信息公开而未完成。
- 正式网站仍运行 `v304`，结论继续隐藏。

推荐的下一步不是直接公开任意长文本，而是确定以下之一：

1. 标准技术术语白名单和规范化映射。
2. 恢复授权访问后展示完整脱敏结论。
3. 继续隐藏结论，仅公开聚合统计。

## 4. 审计门禁

任何公开发布必须满足：

- 输入只能是 `dashboard_ui/demo_data/data_bundle.js`。
- 预写出审计没有发现已知标识或敏感模式。
- 发布目录二次扫描无泄露候选。
- `privacy_audit.json` 的 `status` 为 `pass`。
- 不包含原始 Excel、真实路径、签名、真实登记号或内部审计日志。

## 5. 风险解释

标准医学术语本身通常不具备直接识别性；风险主要来自它与年龄段、性别、型号、植入时间、历次程控时间线等组合后的病例画像。因此需要区分：

- 直接标识删除；
- 假名化病例；
- 真正不可合理重新识别的匿名化公开数据。

用户确认并不能替代机构伦理、数据所有权和数据治理审查。
