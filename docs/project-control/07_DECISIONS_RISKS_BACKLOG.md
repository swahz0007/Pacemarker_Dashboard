# 决策、风险与临床优化迭代路线图

更新时间：2026-07-14  
当前阶段：离线临床工作台优化期  
当前基线：科室真实电脑、真实目录结构、完全离线环境验证通过

## 1. 当前结论

2026-07-14，离线版已在科室工作电脑完成实际试用，程序启动、数据目录选择、全量处理、页面查询和离线运行均稳定，科室电脑不需要安装 Python，也不依赖互联网。项目的技术基座可以视为正式落地，后续工作从“证明能运行”切换到“让医生查得更快、看得更舒服、异常更透明”。

本文件是当前唯一的优化迭代主清单。每轮只解决一个清晰主题，完成本机验证和科室验证后再关闭任务，避免同时大改数据、界面和打包流程。

当前主线是**离线临床查询工作台**。公开演示站点与 SCI 数据分析继续保留，但作为相互隔离的支线，不与科室真实病例数据、人工核查状态或临床版发布节奏混用。

## 2. 不变原则

1. **原始报告是事实源。** 程序默认只读 NAS/文件夹中的 Excel，不静默修改原始报告。
2. **矛盾必须可见。** 未匹配、提取失败、身份冲突和关键字段异常不得被安静忽略，也不得混入正常统计后不作提示。
3. **不自动猜测临床结论。** 系统负责提取、组织、检索和提示；临床异常规则必须经过医生确认，程序不能自行“纠正”医学数值。
4. **上一版可用结果优先。** 新一轮处理失败时保留上一次完整可用数据，不能生成半套结果覆盖稳定版本。
5. **视觉层级服务阅读。** 颜色、字号和粗细表达信息角色，不依靠纯装饰，也不只靠颜色传递风险。
6. **小步发布。** 每个迭代均经历“开发检查 → 本机实测 → 科室实测 → 验收记录”，通过后再进入下一项。
7. **离线边界不改变。** 正式离线包仍直接生成在项目根目录的 `PacemakerDashboard_offline/` 文件夹，不生成 ZIP，不包含患者数据和本机配置。

## 4. I-001：数据质量与人工核查中心（P0）

### 4.1 目标

在 Dashboard 中增加一块独立的“数据质量与人工核查”区域，把当前藏在日志、CSV 和 JSON 中的异常汇总成医生可读的列表。它首先是**只读核查工具**，不在第一版中直接改写原始报告。

现有基座已经具备：

- 模板匹配报告：`matching_report.csv`；
- 身份矛盾隔离队列：`quarantine_records.json`；
- 未匹配模板时停止全量覆盖并保留既有患者输出；
- 身份冲突隔离自动化测试。

因此第一轮的核心是汇总、展示和追溯，而不是重做提取流程。

### 4.2 异常分类

| 级别 | 类型 | 示例 | 默认处理 |
|---|---|---|---|
| 阻断 | 模板未匹配、关键提取失败、输出不完整 | 新格式报告无法识别 | 停止覆盖稳定数据，并明确提示 |
| 待核查 | 身份信息矛盾 | 文件名姓名与报告内姓名不一致 | 隔离，不进入正常患者汇总 |
| 警告 | 关键字段缺失或格式异常 | 日期无效、单位缺失、关键值无法解析 | 保留原值与来源，等待人工确认 |
| 提示 | 可疑重复或非标准字段 | 同一来源重复、厂商新增字段 | 不直接判错，列入观察 |

“临床数值超范围”暂不自动归为异常。只有在规则、单位、设备类型和适用范围经医生确认后，才单独增加临床规则检查，避免误报。

### 4.3 页面内容

顶部统计卡至少显示：

- 本次处理时间、当前数据源路径、扫描文件总数；
- 匹配成功、未匹配、提取成功、提取失败；
- 正常入库患者数、隔离记录数；
- 待核查、已确认、已解决数量；
- 当前 Dashboard 使用的是本次结果还是上一次稳定结果。

核查列表每条至少包含：

- 严重级别和问题类型；
- 患者姓名/登记号（能可靠提取时）；
- 报告日期、设备厂商/型号（能可靠提取时）；
- 来源文件名与完整路径；
- 发生矛盾的字段、两个原始值及其来源；
- 系统已采取的动作，例如“已隔离，未计入患者统计”；
- 人工状态、备注、处理时间。

提供按“问题类型、严重级别、处理状态、日期”筛选，支持患者姓名/登记号搜索，并提供“复制来源路径”或“打开所在文件夹”。页面统计数字必须能下钻到对应明细，不能出现“显示 5 条，列表只有 4 条”的口径差异。

### 4.4 人工核查流程

推荐默认流程：

1. 医生在异常中心发现待核查记录；
2. 通过来源路径打开原始报告并核对；
3. 有权限时修正原始 Excel；无权限时先记录核查结论；
4. 重新执行数据处理；
5. 系统识别问题是否消失，并保留一条可追溯的处理记录。

人工状态建议使用：`待核查`、`已确认原报告无误`、`源文件已修正`、`暂时忽略`。选择“暂时忽略”必须填写原因。

第一版的核查状态只保存在当前电脑的 `%LOCALAPPDATA%\PacemakerDashboard`，不写入 NAS。若以后需要多台电脑共享核查进度，再单独评估共享文件锁、人员标识和审计要求。

不建议直接编辑自动生成的患者 JSON；这会让修改无法回溯，也可能在下次处理时被覆盖。若实际工作证明原始报告不能修改，再设计一个有原值、覆盖值、修改人、时间和原因的受控修正层。

### 4.5 分步实现

- [x] I-001A：只读统计卡和异常明细，接入现有匹配报告与隔离队列；本机样本和 EXE 构建已通过，待科室人工测试。
- [x] I-001B：增加处理状态筛选、搜索、来源追溯和本机核查状态/备注；本机通过，待科室人工测试。
- [ ] I-001C：增加“问题已消失/仍存在”的复跑对照与核查历史。
- [ ] I-001D：仅在确有需求时评估受控修正层，不预先建设。

### 4.6 验收标准

- 构造 0、1、多个身份冲突时，统计卡与隔离文件数量完全一致。
- 未匹配模板、提取失败和身份冲突均有独立入口，不能只出现在终端日志中。
- 每条问题能追溯到唯一来源文件和原始字段值。
- 隔离病例不进入正常患者数、图表和查询结果。
- 修正源文件并复跑后，问题从待核查列表正确转为已解决。
- 核查操作不改变原始 Excel；无互联网、无 Python 环境下仍可使用。

## 5. I-002：突出 `[!] 事件与结论`（P1）

### 5.1 设计方向

把事件与结论从普通键值列表提升为详情页的核心阅读卡，建议放在患者基本信息之后、参数明细之前，使医生打开单次程控记录后无需向下寻找结论。

卡片内部按以下顺序展示：

1. **核心结论**：最大字号和最高字重，完整换行；
2. **需关注事件**：AT/AF、VT 等按项目分组，不挤成一行；
3. **随访建议/下次随访**：与结论分区，避免混读；
4. **其他原始文本**：存在时放在次级区域，不抢占主结论层级。

普通结论使用浅色主题中的高对比中性色；确需复核使用琥珀色；只有规则明确的高风险提示才使用红色。颜色旁必须同时有文字标签或图标，例如“需核查”，不能只靠红/黄区分。

### 5.2 验收标准

- 打开记录后 3–5 秒内能定位核心结论、关键事件和随访时间。
- 结论为空、短句、65 字长句和多行文本均不遮挡、不截断。
- 结论强调不改变原始文字，不用关键词猜测风险级别。
- 1366px 科室常用屏幕、1920px 屏幕和 390px 移动视口均完整显示。

### 5.3 当前实现

- 结论卡提升为详情页首要阅读卡，核心结论与下次随访分区显示并支持长文本换行。
- `[!]` 事件区改为独立全宽卡片，AMS、AT/AF、VT 和其他事件按项目分组展示。
- 普通事件使用中性浅色层级，不根据原始文本猜测临床风险；保留原始内容和“无”状态。
- 本机脚本检查、Python 单元测试和 EXE 构建已通过，等待科室人工测试。

## 6. I-003：重排 `[B] 电池与起搏参数`、`[V] 阈值测试`（P1）

### 6.1 阅读结构

不再把所有原始字段连续铺成同一种键值行，而是按临床阅读顺序分组：

**设备概览**

- 电池状态、电池电压、预计剩余寿命；
- 起搏模式、基础频率、最大跟踪/传感频率；
- 关键的起搏比例。

**分腔参数矩阵**

| 项目 | 心房 RA | 右室 RV | 左室 LV |
|---|---|---|---|
| 感知 | 值 + 单位 | 值 + 单位 | 值 + 单位 |
| 阻抗 | 值 + 单位 | 值 + 单位 | 值 + 单位 |
| 阈值 | 幅度 / 脉宽 | 幅度 / 脉宽 | 幅度 / 脉宽 |
| 输出设置 | 幅度 / 脉宽 | 幅度 / 脉宽 | 幅度 / 脉宽 |

仅展示设备实际存在的腔室；未知或缺失值统一显示 `—`，不得用 `0` 代替。数值、单位和小数点尽量对齐，参数值使用易比较的等宽数字。手机或窄窗口改为按腔室纵向卡片，不强行压缩三列。

无法归类的新厂商字段不能丢弃，应折叠到“其他原始参数”，为后续模板迭代保留线索。任何异常高亮都必须建立在已确认的设备类型、字段含义和医学规则上。

### 6.2 当前实现

- `[B]` 区域按实际模板字段展示电池电压、预估寿命、起搏模式和上下限频率，不派生未出现在模板中的“报告状态”。
- `[V]` 区域以腔室为行展示阻抗、感知、阈值和输出，仅渲染报告实际提供数据的腔室。
- 无可展示导线数据时显示明确空状态；未提供字段统一显示 `—`，不以 `0` 代替。
- 本机脚本检查、Python 单元测试和 EXE 构建已通过，等待科室人工测试。

### 6.3 验收标准

- 医生无需在长列表中来回扫视即可找到电池寿命、模式和各腔阈值。
- 单位始终与对应数值绑定，不跨行、不串列。
- 单腔、双腔、三腔设备及字段缺失记录均能自然排版。
- 原始提取字段总量在重排前后可核对，未识别字段进入“其他原始参数”。
- 最长字段名、较长数值和 125%/150% Windows 缩放下不重叠。

## 7. I-004：分区字体与显示设置（P1）

### 7.1 两层控制

第一层是开发侧的语义样式变量，避免以后为了改一个结论字号而误伤全站。至少拆分：

- 页面正文、导航、患者列表；
- 区块标题、参数标签、参数数值；
- 核心结论、状态徽标、辅助说明；
- 正常、需核查、阻断三类语义颜色。

第二层是程序内的“字体与显示”设置，允许用户按区域微调：

- 患者列表字号；
- 参数标签字号与粗细；
- 参数值字号与粗细；
- 核心结论字号、粗细和强调色；
- 界面密度：紧凑、标准、大字；
- 一键恢复默认。

设置只保存到浏览器本地，不进入患者数据。第一版使用经过验证的字号阶梯、字重和浅色调色板，不开放任意颜色选择器，避免低对比度或界面失去一致性。像素风格继续保留，但文字辨识度优先于装饰效果。

### 7.2 验收标准

- 调整一个区域不会改变其他区域的字号或布局。
- 设置在程序重启后仍保留，并可一键恢复。
- 三档密度在 1366px、1920px、125%/150% 系统缩放下无截断。
- 风险状态除颜色外同时有文字或图标标识。

### 7.3 当前实现

- 侧栏新增“字体与显示”入口，设置面板提供界面密度、患者列表字号、参数标签/数值字号、核心结论字号/字重/颜色等受控预设。
- 设置仅保存到当前浏览器的 `localStorage`，不会写入患者数据、原始 Excel 或离线包配置；支持一键恢复默认。
- 显示变量按患者列表、参数区域、核心结论和报告分区分别作用，改变一个区域不会整体放大页面。
- 已通过 JS 语法检查、Python 单元测试和离线 EXE 构建；待本机人工确认重启后保留、三档密度和 125%/150% 缩放表现。

## 8. I-005～I-007：后续效率与稳定性优化

### I-005：临床查询效率

- 姓名、登记号、设备型号和厂商的统一搜索；
- “仅看最近一次程控”“按时间看纵向变化”的明确切换；
- 待核查、近期新增、存在关键事件等快捷筛选；
- 患者详情返回列表时保留原筛选和滚动位置；
- 搜索结果中直接显示最后程控日期和一行核心结论。

是否增加电池关注、阈值变化等医学筛选，必须先由医生确认字段与规则。

### I-006：更新与运行状态

- 首页或启动结果显示数据源路径、上次成功更新时间和程序版本；
- 明确显示本轮新增、更新、删除、隔离、失败数量；
- 数据处理失败时说明当前页面仍使用哪一次稳定结果；
- 提供查看日志、匹配报告和核查中心的清晰入口；
- 支持重新选择数据目录，但不把某台电脑的路径写进离线包。

### I-007：自动化回归

- 后端：未匹配模板、身份冲突、增量更新、删除源文件、处理失败保留旧数据；
- 前端：长结论、空字段、多腔参数、未知字段、筛选与核查数量一致；
- 视觉：1366×768、1920×1080、390px，以及 125%/150% 缩放；
- 离线：无网络、无 Python、全新电脑、不同数据路径、U 盘复制；
- 安全：确认原始 Excel 未被程序写入，离线包中不存在患者数据和本机配置。

建议的数据流保持单向：

人工核查状态与患者提取数据分开保存，防止重新处理数据时丢失状态，也避免人工备注污染原始临床字段。

## 10. 每轮迭代的完成标准

一项任务只有同时满足以下条件才标记“完成”：

1. 功能按本文件的验收场景实现；
2. 相关自动化测试通过，未破坏现有数据安全保护；
3. 在开发环境完成典型与极端数据检查；
4. 重新生成根目录下的 `PacemakerDashboard_offline/` 文件夹，不生成 ZIP；
5. 在用户自己的电脑完成全新运行或升级运行测试；
6. 在科室电脑完成实际查询验证；
7. 将遇到的问题、决定和验收结论回写本文件或项目变更记录。

状态统一使用：`待开始`、`进行中`、`本机通过`、`科室通过`、`完成`、`暂停`。

## 11. 已采用的默认决策与待确认项

### 已采用的默认决策

- 离线临床工作台是当前产品主线；公开演示与 SCI 为隔离支线。
- 首版异常中心只读展示，不直接修改源报告。
- 首版核查状态保存在当前电脑，不写 NAS、不跨电脑共享。
- 原始报告可修改时优先修正事实源；受控覆盖层延后评估。
- 参数异常规则必须经医生确认后才能上线。
- 浅色像素风继续作为默认视觉方向，但临床可读性优先。

### 后续需要在真实使用中回答

- “矛盾病例”除身份冲突外，还应包含哪些明确、可验证的规则？
- 科室是否需要多人或多电脑共享核查状态？
- 哪些原始报告没有修改权限，是否确实需要受控修正层？
- 哪些电池、阈值或事件条件应被标记为“需关注”，适用哪些厂商与设备？
- 字体设置使用三档预设是否足够，还是确有按区域高级定制需求？

## 12. 风险清单

| 风险 | 影响 | 控制措施 |
|---|---|---|
| 异常统计漏项或口径不一致 | 医生误以为数据完整 | 总数可下钻、固定分母、自动化对账 |
| 将正常差异误判为临床异常 | 增加误报甚至干扰判断 | 规则需医生确认，保留原值与适用范围 |
| 人工修改生成文件 | 下次处理被覆盖且无法追溯 | 优先修改事实源，人工状态与提取数据分离 |
| 结论过度使用红色 | 视觉疲劳、风险含义失真 | 红色只用于确认的高风险，普通结论用中性强调 |
| 参数重排时遗漏厂商字段 | 信息丢失 | 未识别字段进入“其他原始参数”，做字段总量核对 |
| 字体自由度过高 | 界面混乱或低对比 | 使用语义变量、受控预设和一键恢复 |
| 多电脑核查状态不一致 | 重复核查或结论不同步 | 首版明确本机状态；共享需求另立数据模型 |
| 本地、科室与公开版本分叉 | 误把某版本能力当成全部已上线 | 页面显示版本与数据时间，三条发布线分别记录 |
| 工作树大量未提交变更 | 难以回滚和定位回归 | 每轮限定改动范围，测试后按主题审查提交 |
