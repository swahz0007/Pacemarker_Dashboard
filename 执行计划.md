# Pacemaker Dashboard 科室离线部署与临床场景试用执行计划

> 制定日期：2026-07-13  
> 当前状态：离线 EXE 原型已构建，待用户电脑接入 NAS 验证  
> 建议周期：约 4–6 周（含 2–4 周影子试用）

## 一、结论

**可行，而且现有项目已经具备大部分技术基础。**

推荐的落地方式不是把整个开发仓库和真实报告一起复制到普通电脑上直接运行，而是制作一个受控的“科室离线运行包”：程序以**只读方式**扫描科室现有程控报告目录，把提取结果写入独立的受保护工作目录，再由本机浏览器打开 Dashboard 查询。

第一阶段建议限定为：**一台科室授权电脑、1–2 名授权使用者、全程离线或仅处于医院可控内网，直接启用并在前 1–2 周边用边抽查**。Dashboard 用于检索、汇总和辅助查看；涉及患者诊疗的关键结论仍以原始程控报告、设备状态和临床判断为准。

现阶段不建议未经改造就直接投入使用，必须先完成以下 P0 项：

1. 把固定数据路径改为可配置的科室报告目录；
2. 确保原始报告目录只读，所有派生数据写入独立目录；
3. 增加共享盘断连、扫描数量骤降和误删除保护；
4. 准备完整的离线依赖与一键更新/启动流程；
5. 完成真实目录结构兼容测试、字段抽查和隐私安全验收；
6. 明确使用权限、操作留痕、故障停用和回退机制。

## 二、现有基础与主要缺口

| 项目 | 当前基础 | 离线临床版需要补齐 |
|---|---|---|
| 报告扫描 | 已递归扫描子目录，支持 `.xls` / `.xlsx`，跳过 `~$` 临时文件 | 数据根目录目前固定为项目内 `01_data_repository/`，需支持科室真实路径或 UNC 共享路径 |
| 数据处理 | 已有模板匹配、字段提取、患者归并、身份冲突隔离和独立审计 | 需用科室真实模板做覆盖测试，并让所有异常在界面或运行摘要中可见 |
| 增量更新 | 已按 SHA-256 识别新增、修改和删除 | 共享盘部分断连可能被误判为删除，临床版必须默认阻断批量删除并人工确认 |
| 输出安全 | 患者 JSON 原子写入；全量失败时保留既有输出 | 输入、输出必须物理分离；增加并发锁、空间检查和单一受控回退点 |
| 前端离线能力 | Chart.js 已本地打包；`file://` 模式可加载本地 `data_bundle.js` | 当前本地 bundle 含完整病例数据，必须依靠受控电脑、磁盘加密和目录 ACL 保护 |
| 使用体验 | 已可按患者和字段查看 Dashboard | 需增加“科室离线模式”、数据更新时间、扫描状态、未匹配和隔离记录提示 |
| 部署 | 开发环境可用 Python/Node 运行 | 科室电脑不应现场联网安装；需准备经校验的离线 Python 依赖和预构建前端资源 |
| 权限与审计 | 当前公开演示入口默认放行，不适合真实病例 | 单机试用至少依赖 Windows 授权账户；多人/内网服务化前必须增加用户权限和访问留痕 |

## 三、推荐目标架构

```text
科室程控报告目录（只读、唯一事实来源）
                │
                ▼
本机离线扫描 → 模板匹配 → 字段提取 → 身份冲突隔离 → 完整性检查
                │
                ▼
受保护工作目录（patient_records / 索引 / 日志 / 单一回退点）
                │
                ▼
内部 Dashboard 数据包（敏感数据，不上传、不外发）
                │
                ▼
授权电脑本地浏览器查询
```

目录原则：

- **原始报告目录**：程序账户仅授予读取权限，禁止程序改名、移动、覆盖或删除原始 Excel。
- **程序目录**：只放运行代码和已验证依赖，不放真实报告、科研文档、Git 历史、公开演示包或部署凭据。
- **工作目录**：存放结构化患者记录、索引、运行状态和审计结果，按真实临床数据同等级保护。
- **网络边界**：不调用 Netlify、GitHub、CDN、云 API 或外部统计服务；试用期不开放局域网端口。
- **首选运行模式**：单机 `file://` 本地页面，实施最简单、攻击面最小。若以后数据量或多人使用需要服务化，再单独设计仅监听 `127.0.0.1` 或医院内网的受控服务。

## 四、分阶段执行计划

### 阶段 0：合规确认与环境盘点（1–3 个工作日）

任务：

- 由科室负责人、信息科/网络安全部门确认用途、使用范围、数据目录、使用人员和设备。
- 明确本项目第一阶段是“病例检索与研究/工作辅助”，不是自动诊断或独立临床决策系统。
- 确认科室报告目录的实际路径、是否为本地盘或共享盘、目录层级、文件总量、历史年份、品牌/设备类型和 `.xls` / `.xlsx` 比例。
- 确认目标电脑的 Windows 版本、64 位架构、浏览器、可用磁盘空间、Python 安装权限、磁盘加密、账户权限和备份策略。
- 确认电脑不启用个人网盘同步、浏览器账号同步和非必要扩展，不允许把工作目录复制到个人 U 盘或外部设备。
- 记录当前代码基线、依赖版本和校验值，避免带着未确认的开发变更进入科室环境。

阶段产物：经确认的部署范围、设备与路径清单、授权用户清单、数据处理与故障上报责任人。

通过门槛：**在机构授权、设备边界和数据用途未明确前，不导入真实报告。**

### 阶段 1：离线临床版改造（约 3–5 个工作日）

#### 1. 路径与配置

- 在 `backend/config.py` 中增加外部报告目录和工作目录配置，例如环境变量 `PACEMAKER_REPORT_DIR`、`PACEMAKER_WORK_DIR`。
- 保留现有项目内目录作为开发默认值，科室版通过本机配置覆盖，不把单位真实路径写入仓库。
- 启动前校验：输入目录存在且可读、输出目录可写、输入和输出不能互相包含、剩余空间足够、路径未指向公开同步目录。
- 支持中文路径、空格路径和医院 UNC 共享路径。

#### 2. 原始数据只读保护

- 数据扫描流程只调用读取和哈希操作，不在报告目录生成缓存、日志或索引。
- 增加只读自检：发现程序对源目录具有非预期写入行为时立即终止。
- 继续跳过临时文件、损坏文件和不支持格式，并把问题列入可见的待处理清单，禁止静默遗漏。

#### 3. 增量更新安全闸门

- 报告目录不可访问、子目录访问异常或扫描未完整完成时，**不执行任何删除同步**。
- 试用期默认只处理新增和修改；删除来源先标记为“待确认”，不立即移除患者记录。
- 若本次可见文件数较上次异常下降，立即停止更新并保留既有数据；阈值在科室基线盘点后确定。
- 增加单实例运行锁，防止重复点击导致两个更新任务同时写入。
- 更新成功后再原子替换正式索引；失败时继续使用上一次完整可用版本。

#### 4. 内部 Dashboard 构建

- 沿用 `dashboard_ui/scripts/generate_data.py` 生成仅供本机使用的真实数据包，严禁把该数据包用于公开构建或 Netlify 发布。
- 页面明显标识“科室离线模式”“数据更新时间”“本次扫描文件数”“未匹配数”“隔离记录数”。
- 当存在未匹配模板、提取失败、身份冲突或审计未通过时，显示醒目告警；不能把“页面能打开”等同于“数据完整”。
- 保留风险提示：Dashboard 是辅助查询结果，关键诊疗信息必须回看原始报告。

#### 5. 一键操作与日志

- 实施时优先复用现有脚本结构；确有必要时，经确认后新增“更新本地数据”和“启动 Dashboard”两个入口。
- 更新入口按固定顺序执行：环境检查 → 增量提取 → 完整性检查 → 生成 Dashboard 数据 → 显示摘要。
- 普通运行摘要不显示患者姓名、登记号或完整源路径；详细错误日志只保存在受控工作目录，禁止外发。
- 日志至少记录版本、时间、操作者/Windows 账户、扫描数、新增数、修改数、待确认删除数、失败数和最终状态。

### 阶段 2：制作并验证离线 EXE 运行包（约 2–3 个工作日）

当前已采用 PyInstaller 制作目录版 EXE 原型。运行包应只包含实际运行所需内容：

- `PacemakerDashboard.exe` 及其 `_internal/` 运行资源；
- 已预构建的 Dashboard 页面、Chart.js 和模板定义；
- 版本说明、文件校验值和最小化操作说明；
- 构建端单独保留 `requirements-build.txt` 和 PyInstaller 配置，不能复制到科室作为源代码运行环境。
- 版本说明、文件校验值和最小化操作说明。

EXE 原型运行时由用户选择 NAS 报告目录，默认把派生数据写入 `%LOCALAPPDATA%\\PacemakerDashboard\\runtime_data`。科室电脑只需要 EXE 运行包，不需要安装 Python；开发端通过 `build_offline_exe.ps1` 构建，输出为项目根目录下的 `PacemakerDashboard_offline/` 文件夹。交付时直接复制整个文件夹，不再生成 ZIP。

运行包不得包含：

- `01_data_repository/`、`patient_records/`、`dashboard_ui/data/` 中的现有真实数据；
- `.git/`、`99_backup/`、SCI 论文资料、开发缓存、个人配置；
- Netlify/GitHub 配置凭据、token、公开演示构建产物；
- 与科室运行无关的大型 PPT、图片生成物和开发依赖。

构建端验证：

```powershell
python -m pip install --no-index --find-links <离线依赖目录> -r requirements-build.txt
python -m unittest discover -s tests -p "test_*.py"
```

科室端不安装 Python 或 Node.js，只需在无互联网连接的 Windows 环境中运行 EXE，走通选择 NAS 目录、扫描、更新、打开 Dashboard 和回退流程，再复制到单位。

### 阶段 3：技术验收与真实数据小样本核对（约 2–4 个工作日）

先用脱敏/模拟目录验证，再在获批环境中使用真实报告。测试矩阵至少覆盖：

- 每个仍在使用的品牌、设备类型和模板；
- `.xls`、`.xlsx`、中文路径、深层子目录、空目录；
- 临时文件、损坏文件、被 Excel 占用的文件和未知模板；
- 同名患者、登记号冲突、同一患者多次程控、文件重命名；
- 新增、修改、待确认删除、共享盘断连和部分目录无权限；
- 首次全量处理、无变化重复运行和少量文件增量更新；
- 完全断网时页面、搜索、详情和图表是否正常。

人工核对建议采用“模板分层 + 随机抽样”：每个实际模板至少抽取若干份报告，再随机抽取 20–30 份纵向记录，逐项核对姓名/登记号、品牌型号、日期、电池、导线参数、事件、结论及程控次数。具体样本量由科室和信息科根据风险决定，不预设提取准确率为 100%。

验收门槛：

- 原始报告目录零写入；
- 手工基线文件数与程序发现数一致，所有差异均有明确解释；
- 未匹配模板为 0，或已明确隔离且不会出现在正式查询结果中；
- 身份冲突不会自动合并到错误患者；
- 关键字段不存在未解释的不一致，审计问题已分类并有处理决定；
- 断网状态下全部核心功能可用，未发现外部网络请求；
- 更新中断后旧版 Dashboard 仍可用；
- 搜索和患者详情在当前数据规模下达到日常可接受速度。建议先以页面查询 2 秒内返回作为试运行目标，再根据实测调整。

### 阶段 4：科室直接启用与短期观察（1–2 周）

启用范围：

- 仅部署在一台受控电脑；
- 仅开放给 1–2 名指定人员；
- 每日首次使用前执行一次增量更新，新增报告较多时按需再次更新；
- Dashboard 从第一天即可用于快速查询，但前 1–2 周对异常结果和少量随机记录回看原始报告确认。

试用期间记录以下非患者标识化问题：

- 找不到应存在的报告；
- 患者错误归并或同一患者未归并；
- 字段提取错误、缺失或单位异常；
- 搜索条件、展示方式或操作步骤不符合实际工作流；
- 更新耗时、页面响应、共享盘稳定性问题；
- 未匹配、隔离或审计告警的处理耗时。

问题统一更新到现有 `docs/project-control/07_DECISIONS_RISKS_BACKLOG.md`，避免另建零散版本文件或包含患者身份的问题截图。

立即停用条件：

- 发现跨患者错误合并；
- 发现程序写入或改动原始报告；
- 出现数据外传、未授权访问或敏感文件进入同步盘；
- 目录异常下降但程序仍自动删除正式记录；
- 关键字段出现可能影响工作判断的系统性错误；
- 无法确认当前页面数据是否为最新完整版本。

### 阶段 5：评估、固化与后续迭代（约 2 个工作日，之后持续）

短期观察结束后召开一次 Go / No-Go 评审：

- 汇总查询次数、节省时间、发现的问题、误差类型和用户建议；
- 确认是否继续单机使用、扩大到更多用户，或暂缓并修正；
- 将已验证需求按 P0/P1/P2 排序，每次只发布一个可回退的小版本；
- 每次升级在外部开发环境完成，使用模拟/脱敏数据测试，通过校验后再以离线介质导入；
- 单位内不得导出真实数据供远程调试。确需分析问题时，只提供脱敏样例、错误类别和最小复现信息。

如以后扩展到多电脑或医院内网访问，必须另行完成：应用级身份认证、最小权限、逐次访问日志、超时退出、内网 TLS、信息科安全评估和正式运维责任划分；不能直接把当前静态目录共享给多人使用。

## 五、建议的日常操作流程

1. 使用个人授权 Windows 账户登录科室电脑。
2. 确认科室报告目录正常可访问。
3. 运行“更新本地数据”，查看扫描、成功、失败、未匹配、隔离和待确认删除摘要。
4. 只有状态为“完整可用”时才打开 Dashboard；异常时继续使用上一次完整版本并处理告警。
5. 在 Dashboard 中按姓名、登记号、设备或随访信息查询。
6. 普通检索可直接使用 Dashboard；关键诊疗信息或异常结果回看原始程控报告。
7. 发现问题时停止依赖该字段，记录无直接标识的问题描述，并进入统一待办。
8. 使用结束后关闭页面并锁定电脑，不复制内部 bundle、患者 JSON 或截图到个人设备。

## 六、实施任务优先级

| 优先级 | 任务 | 主要涉及位置 | 完成标准 |
|---|---|---|---|
| P0 | 外部报告目录与独立工作目录配置 | `backend/config.py` | 无需移动原始报告即可扫描，真实路径不进入仓库 |
| P0 | 共享盘断连与批量删除保护 | `backend/core/file_tracker.py`、`backend/main.py` | 不完整扫描不改正式输出，删除需确认 |
| P0 | 离线数据更新和完整性摘要 | `backend/main.py`、`dashboard_ui/scripts/generate_data.py` | 一次操作生成可查询数据并明确报告异常 |
| P0 | 本地模式与风险告知 | `dashboard_ui/index.html`、`dashboard_ui/assets/js/app.js` | 页面显示本地状态、版本、更新时间和告警 |
| P0 | 路径、删除、断连、冲突回归测试 | `tests/test_pipeline_safety.py` | 所有安全门禁有自动化测试 |
| P0 | 离线依赖与干净机验证 | `requirements.txt`、离线运行包 | 无网络可安装、测试和运行 |
| P1 | 一键更新、一键启动和运行锁 | 现有批处理/启动流程 | 普通使用者无需命令行，重复点击不会并发写入 |
| P1 | 访问与更新操作留痕 | 本地受控日志 | 可追溯版本、时间、操作者和处理摘要 |
| P1 | 查询体验迭代 | Dashboard UI | 根据真实工作流优化搜索和筛选 |
| P2 | 分记录按需加载 | Dashboard 数据层 | 数据量增长后仍能快速启动，避免一次加载全部病例 |
| P2 | 多用户内网版 | 独立立项 | 完成认证、授权、审计和信息科安全验收后再实施 |

## 七、关键风险与控制

| 风险 | 后果 | 控制措施 |
|---|---|---|
| 把“离线”误认为“天然安全” | 本地复制、截图、同步盘或共用账户仍可泄露 | 受控设备、磁盘加密、Windows 账户、ACL、禁用同步、最小权限 |
| 共享目录短暂断连 | 大量报告被误判删除 | 扫描完整性门禁、数量骤降阻断、删除待确认、旧版继续可用 |
| 新模板或报告格式变化 | 静默漏提取或字段错位 | 未匹配 fail-closed、模板分层抽查、独立审计、告警可见 |
| 同名/登记号冲突 | 跨患者错误归并 | 延续隔离队列，禁止按文件名强制合并，人工复核 |
| 本地 bundle 含完整病例 | 获得文件即可读取全部数据 | 单机试用、受控工作目录、禁止外发；后续改为逐例加载和访问审计 |
| 软件结果被当作最终临床结论 | 影响诊疗安全 | 明确辅助定位边界，关键字段回看原始报告，异常立即停用 |
| 离线环境难以修复 | 依赖缺失或升级失败 | 完整 wheel 包、版本校验、干净机演练、单一回退点 |
| 开发版与科室版分叉 | 修复状态不清、误覆盖配置 | 统一版本号和变更记录，配置与代码分离，小版本可回退 |

## 八、合规与治理依据

本计划不替代本单位信息科、伦理、数据管理和医疗质量部门的正式审查。医疗健康信息属于敏感个人信息；即使系统不连接互联网，也仍涉及采集、存储、处理、访问控制和留痕责任。

- [《中华人民共和国个人信息保护法》](https://www.miit.gov.cn/jgsj/zfs/fl/art/2022/art_515a4b20c12f430eab54bb4f56d89f56.html)：医疗健康信息属于敏感个人信息，处理应具有特定目的、充分必要性并采取严格保护措施。
- [《医疗卫生机构网络安全管理办法》](https://www.nhc.gov.cn/guihuaxxs/c100133/202208/8a23d01133214a779879094dd20cd383.shtml)：要求明确主管、运营、信息化和使用部门责任，并覆盖数据全生命周期安全、风险评估和应急处置。
- [《关于进一步加强医疗机构电子病历信息使用管理的通知》](https://www.nhc.gov.cn/yzygj/c100068/202506/c68abee7c54b4651a774cd533761780b.shtml)：强调最小可用权限、专有身份标识、操作可追溯和异常访问监测。
- [《医疗机构临床决策支持系统应用管理规范（试行）》](https://www.nhc.gov.cn/wjw/c100378/202307/5474ed8d51344b4a95bb10fb29a6d0ff.shtml)：若未来系统功能扩展到临床决策支持，应进一步落实风险告知、使用日志、权限审计和可控内部网络安全要求。

## 九、首轮实施完成定义

只有同时满足以下条件，才算完成“可进入科室直接启用”的首轮交付：

- [ ] 机构、科室和信息科确认试用范围与责任人；
- [ ] 程序可只读扫描科室真实目录，无需复制或改动原始报告；
- [ ] 所有结构化输出只进入受控工作目录；
- [ ] 共享盘断连、数量骤降和删除保护测试通过；
- [ ] 完整离线依赖包在干净电脑安装验证通过；
- [ ] 真实模板覆盖测试、人工抽查和独立审计完成；
- [ ] Dashboard 断网可用，并显示版本、更新时间和异常状态；
- [ ] 一键更新、失败回退和问题上报流程完成演练；
- [ ] 授权用户完成使用边界与数据安全培训；
- [ ] 明确 1–2 周观察期的开始日期、责任人和停用条件。
